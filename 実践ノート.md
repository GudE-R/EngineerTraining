## 03/08

### RLS（Row Level Security）× StudyTodo — 実際に踏んだバグから学ぶ

#### 何が起きたか

モバイル版でログインしたら、こんなエラーが出た：
```
Sync Error: {"code": "42501", "message": "new row violates row-level security policy (USING expression) for table \"srs_profiles\""}
```
「srs_profilesテーブルへの書き込みがRLSに弾かれた」というエラー。

#### RLSとは何か

Supabase（PostgreSQL）のセキュリティ機能で、**テーブルの行ごとにアクセス制御**をかける仕組み。

StudyTodoの `srs_profiles` テーブルにはこんなポリシーが設定されている：
```sql
CREATE POLICY "Users can CRUD their own SRS profiles"
  ON srs_profiles FOR ALL
  USING (auth.uid() = user_id)          -- 読み取り・更新・削除の条件
  WITH CHECK (auth.uid() = user_id);    -- 挿入・更新時の条件
```

- **USING句**: SELECT / UPDATE / DELETE 時に「既存の行」に対してチェック
- **WITH CHECK句**: INSERT / UPDATE 時に「新しい行」に対してチェック
- `auth.uid()`: 今ログインしているユーザーのID

→ つまり「自分のデータしか読めない・書けない」というルール。

#### なぜエラーになったか — 時系列で追う

1. ユーザーAでログイン → 同期 → ローカルSQLiteにAのデータが保存される
2. ログアウト（**ローカルDBはそのまま残る**）
3. ユーザーBでログイン → 同期が走る
4. 同期ロジックの流れ：
   - クラウドから `SELECT *` → RLSによりBのデータだけ返る
   - ローカルDBにはAのデータも残っている
   - Aのデータは「ローカルにだけ存在する」と判定される
   - Aのデータを `user_id = B` として `upsert` しようとする
5. `upsert` = `INSERT ... ON CONFLICT DO UPDATE`
   - 同じIDの行がSupabase上に存在する（Aのデータとして）
   - UPDATE が走る → USING句で既存行の `user_id`（A）と `auth.uid()`（B）を比較
   - **一致しない → RLS違反でエラー**

```
ユーザーA → ログイン → ローカルDBにAのデータ保存
         → ログアウト（ローカルDBはそのまま！）
ユーザーB → ログイン → 同期開始
         → ローカルのAのデータを「新規」と誤判定
         → Aのデータ(ID)をBとしてupsert
         → Supabase上にはAの行が存在 → UPDATE発動
         → USING (auth.uid()=B) ≠ (user_id=A) → 拒否！
```

#### どう直したか

**ログアウト時にローカルDBを全クリアする処理を追加した。**

`apps/mobile/src/hooks/useMobileSync.ts:96-100`
```typescript
} else if (event === 'SIGNED_OUT') {
    setUserId(undefined);
    if (repo.clearAll) {
        await repo.clearAll();
    }
}
```

`apps/mobile/src/repositories/SQLiteRepository.ts:391-397`
```typescript
async clearAll(): Promise<void> {
    await this.db.runAsync('DELETE FROM todos');
    await this.db.runAsync('DELETE FROM categories');
    await this.db.runAsync('DELETE FROM srsProfiles');
    await this.db.runAsync('DELETE FROM sessions');
    await this.db.runAsync('DELETE FROM feedbacks');
}
```

→ ログアウトでローカルDBがリセットされるので、別アカウントのデータが混ざらなくなる。

#### 学びのポイント

| ポイント | 内容 |
|---------|------|
| RLSの2つの句 | USING = 既存行のチェック、WITH CHECK = 新しい行のチェック |
| upsertの罠 | INSERTのつもりでも、IDが既存なら内部的にUPDATEになる |
| ローカルDBの寿命 | ログアウトしてもSQLiteは消えない。明示的にクリアが必要 |
| マルチアカウント設計 | ローカルストレージがアカウントを跨ぐと、IDの衝突・権限エラーが起きる |
| エラーメッセージの読み方 | 「USING expression」→ 既存行のチェックで落ちた ＝ 他人のデータを触ろうとした |
