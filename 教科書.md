# 教科書

# 第1章：数値表現（2進数・補数）
## 1.1 2進数とは
コンピュータはすべての情報を「0」と「1」の2つの状態で管理する。これを2進数と呼ぶ。
- 10進数から2進数への変換：値を2で割り続け、余りを下から並べる。
- 2進数から10進数への変換：各桁に2の累乗（1, 2, 4, 8...）を掛けて足す。

## 1.2 補数と負の数
コンピュータ内で引き算を行うために、「足すと桁が上がる最小の数」を補数として利用する。
- 1の補数：各ビットを反転させる（0を1に、1を0にする）。
- 2の補数：1の補数に1を加える。コンピュータはこの2の補数を使って負の数を表現する。

# 第2章：論理演算
## 2.1 基本的な論理演算
- AND（論理積）：両方が1のときだけ1。
- OR（論理和）：一方が1なら1。
- NOT（否定）：値を反転させる。
- XOR（排他的論理和）：一方が1で、もう一方が0のときだけ1。

# 第3章：データ構造
## 3.1 代表的なデータ構造
- 配列：連続したメモリ領域にデータを並べる。インデックスによる高速なアクセスが可能。
- リスト：データと次のデータへのポインタをセットにする。データの挿入・削除が容易。
- スタック：後入れ先出し（LIFO）。
- キュー：先入れ先出し（FIFO）。
- 木構造：階層的なデータの集まり。

# 第4章：アルゴリズムと計算量
## 4.1 アルゴリズム
- 探索：線形探索、二分探索。
- 整列：バブルソート、選択ソート、クイックソート。
## 4.2 計算量（オーダー）
アルゴリズムの効率を O(n) などの記法で表す。nが大きくなったときの計算時間の増え方を示す。

# 第5章：ネットワーク基礎
## 5.1 通信の仕組み
- プロトコル：通信を行うための共通ルール（HTTP, HTTPSなど）。
- クライアント・サーバーモデル：サービスを要求する側（クライアント）と提供する側（サーバー）。
- ステータスコード：リクエストの結果を示す（200: 成功, 404: 未検出, 500: サーバーエラーなど）。

# 第6章：セキュリティ基礎
## 6.1 セキュリティの3大要素（CIA）
- 機密性 (Confidentiality)：許可された人のみアクセスできる。
- 完全性 (Integrity)：データが正確で、改ざんされていない。
- 可用性 (Availability)：必要なときにいつでも利用できる。
## 6.2 認証と認可
- 認証：本人であることを確認する。
- 認可：アクセス権限を与える。

# 第7章：データベース
## 7.1 リレーショナルデータベース (RDB)
- テーブル：行と列からなる表形式でデータを管理。
- 主キー (PK)：行を一意に識別するための項目。
- 外部キー (FK)：他テーブルとの関連付けを行うための項目。
- 正規化：データの重複を排除し、不整合を防ぐための整理作業。

# 第8章：ソフトウェア開発管理技術（Git/GitHub）

## 8.1 バージョン管理の必要性
システム開発において、ソースコードの変更履歴を管理することは不可欠である。
- **バックアップ**: 過去のどの時点の状態にも戻せる。
- **並行開発**: 複数のメンバーが同時に異なる機能を開発できる。
- **責任の明確化**: 「いつ」「誰が」「なぜ」そのコードを書いたかが記録される。

## 8.2 Gitの基本コンセプト
Gitは「分散型」バージョン管理システムである。
- **リポジトリ**: 変更履歴を保存する場所。
- **ワークツリー**: 実際にファイルを編集している場所。
- **ステージングエリア（インデックス）**: 次回コミット（保存）するファイルを選択する場所。

## 8.3 基本コマンド
1. `git init`: 新しいリポジトリを作成する。
2. `git add <ファイル名>`: 変更をステージングエリアに追加する（録画準備）。
3. `git commit -m "メッセージ"`: 変更をリポジトリに記録する（録画保存）。
4. `git status`: 現在の状態を確認する。
5. `git log`: 過去の履歴を確認する。

## 8.4 GitHubとプルリクエスト
GitHubはGitリポジトリをホスティング（管理）するクラウドサービスである。
- **ブランチ (Branch)**: 履歴の流れを分岐させる機能。`main`（本流）から分岐して作業を行う。
- **プルリクエスト (Pull Request)**: ブランチで行った変更を本流へ合流させるために、他者に「提案・相談」する仕組み。
- **マージ (Merge)**: 分岐したブランチを本流に統合すること。
