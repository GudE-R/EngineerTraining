### [x] 2進数の変換計算が不正確
**10進数→2進数**: 2で割り続けて余りを下から読む。割り算を1ステップずつ丁寧に書くこと。
**2進数→10進数**: 各桁に2の累乗をかけて足す。

必ず暗記すべき2の累乗:
```
2⁰=1  2¹=2  2²=4  2³=8  2⁴=16  2⁵=32  2⁶=64  2⁷=128  2⁸=256
```

検算のコツ: 変換後に逆変換して元の数に戻るか確認する

### [x] 8進数・16進数の対応表が不正確
3桁の対応（8進数）で110=6, 101=5を逆に読むミスあり。
4桁の対応（16進数）で3=0011を0101と間違えるミスあり。

覚え方: 各桁の重みを右から1,2,4（3桁）/ 1,2,4,8（4桁）で足す
```
110 → 4+2+0 = 6
101 → 4+0+1 = 5
0011 → 0+0+2+1 = 3
0101 → 0+4+0+1 = 5
```

### [ ] NOT演算と補数の関係が結びついていない
1の補数を求める「全ビット反転」は、NOT演算そのもの。
- NOT = ひっくり返す = 1の補数
- 補数の「ひっくり返して+1」= NOT + 1 = 2の補数
学んだ概念同士のつながりを意識すること。

### [ ] 2の補数を求めるときのビット反転が不正確
00001010 の反転を 11110110 としてしまい、正しい 11110101 とずれた。

**対策**: 1桁ずつ丁寧にひっくり返す。焦って一気にやらない。
```
元:   0 0 0 0 1 0 1 0
反転: 1 1 1 1 0 1 0 1  ← 1桁ずつ確認
+1:   1 1 1 1 0 1 1 0  ← これが2の補数
```

### [ ] 計算量の表現が曖昧
バブルソートが遅い理由を「計算量が多くなることがある」と答えたが、正確には**常にO(n²)で遅い**。
- 「ことがある」→ 条件次第で速くなる印象を与える
- 実際はデータ数が増えるほど**必ず**二乗で処理回数が増える
- 計算量は「最悪の場合」や「平均」で語る。曖昧な表現は避ける
