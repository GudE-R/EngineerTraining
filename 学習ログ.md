### 2進数・8進数・16進数
- 10進数→2進数: 2で割り続けて余りを下から読む
- 2進数→10進数: 各桁に2の累乗をかけて足す
- 2進数→8進数: 右から3桁ずつ区切って各グループを変換
- 2進数→16進数: 右から4桁ずつ区切って各グループを変換
- 8進数⇔16進数: 2進数を経由して変換する
- 2の累乗は暗記必須: 1, 2, 4, 8, 16, 32, 64, 128, 256

### 補数
- 補数の目的: コンピュータには引き算の回路がないため、足し算だけで引き算を実現する仕組み
- 1の補数: 全ビットを反転（0↔1）
- 2の補数: 1の補数 + 1
- コンピュータは2の補数で負の数を表現する
- 符号ビット: 最上位ビットが0なら正、1なら負
- n の補数 = (n-1)の補数 + 1 という関係が成り立つ

### 論理演算
- AND（論理積）: 両方1のときだけ1
- OR（論理和）: どちらか1つでも1なら1
- NOT（否定）: ビット反転（0↔1）
- XOR（排他的論理和）: 片方だけ1のとき1（両方1は0 ← ORとの違い）
- 複数ビットの演算は各ビットごとに独立して計算する
- 実務用途: AND=マスク処理、OR=フラグ設定、NOT=ビット反転、XOR=値の入れ替え・暗号化

### データ構造
- 配列: 番号（インデックス）で即アクセス。途中への挿入は苦手
- リスト（連結リスト）: 挿入・削除が得意。番号で直接取り出せない
- スタック: 後入れ先出し（LIFO）。push/popの2操作。例: ブラウザの戻る、Ctrl+Z
- キュー: 先入れ先出し（FIFO）。enqueue/dequeueの2操作。例: 印刷待ち
- 木: 階層構造。ルート・ノード・リーフ。例: フォルダ構造、HTML
- データ構造の選び方 = そのデータに対して一番多い操作は何かで決まる

### プログラミング基本文法（言語共通）
- 変数: データに名前をつける箱。型には整数・小数・文字列・真偽値・nullがある
- 演算子: 算術(+ - * / %)、比較(== != < > <= >=)、論理(&& || !)
- 条件分岐: if / else if / else で処理を分ける
- ループ: for（回数指定）、while（条件指定）、forEach（配列の各要素に処理）
- 関数: 処理をまとめて名前をつける。引数（入力）と戻り値（出力）を持つ
- 配列操作: インデックスで取得、push/filter/map
- オブジェクト: 関連するデータをキーと値のペアでまとめる
- === と == の違い: ===は型も値も一致で判定、==は型変換して比較（バグの元）。実務では===を使う（JS/TS特有）

### アルゴリズム
- アルゴリズム = 問題を解くための手順。手順が違えば速さが変わる
- 線形探索: 先頭から順番に探す。ソート不要. O(n)
- 二分探索: 真ん中で半分に切って絞り込む。ソート済み必須. O(log n)
- バブルソート: 隣同士を比べて入れ替え。O(n²)で遅い。教科書向け
- 選択ソート: 最小値を見つけて先頭に置く。O(n²)で遅い
- クイックソート: 基準で2グループに分けて再帰. O(n log n)で実務向き
- 再帰: 自分自身を呼んで問題を小さくしていく。止まる条件が必須
- 計算量: O(1) < O(log n) < O(n) < O(n log n) < O(n²) の順に遅くなる

### ネットワーク基礎
- クライアント（リクエストする側）とサーバー（返す側）の関係
- HTTP = 通信のルール（プロトコル）
- HTTPメソッド: GET=取得、POST=作成、PUT/PATCH=更新、DELETE=削除
- ステータスコード: 2xx=成功、4xx=クライアントミス、5xx=サーバーエラー
- JSON = サーバーとの共通データ形式。TSのオブジェクトとほぼ同じ
- API = サーバーが用意した窓口（REST API: URL+メソッドの組み合わせ）
- トークン認証: ログイン→トークン発行→以降のリクエストにトークンを付ける（入場チケット）
- HTTPS = HTTPを暗号化して安全にしたもの

### セキュリティ基礎
- CIA: 機密性（見せない）・完全性（変えさせない）・可用性（使えるようにする）
- 認証（AuthN）= あなたは誰？ / 認可（AuthZ）= あなたは何ができる？
- SQLインジェクション: 入力にSQLを仕込む攻撃。パラメータ化クエリで防ぐ
- XSS: 入力にスクリプトを仕込む攻撃。エスケープ（無害化）で防ぐ。Reactは自動対応
- CSRF: ログイン中に偽リクエストを送る攻撃。トークン検証で防ぐ
- パスワードはハッシュ化して保存（元に戻せない変換）。Supabase Authが処理
- RLS（Row Level Security）: 行レベルのアクセス制御 = 認可の実装

### データベース（SQL、正規化）
- テーブル=表、行=1件のデータ、列=データの項目
- SQL: SELECT=取得、INSERT=作成、UPDATE=更新、DELETE=削除（HTTPメソッドと対応）
- 主キー（PK）: 行を一意に識別するID。重複しない
- 外部キー（FK）: 別テーブルの主キーを参照して紐付ける
- UUID: ほぼランダムなID。衝突しない。Supabaseではgen_random_uuid()で自動生成
- 正規化: データの重複をなくして整理する。第3正規形まで実施するのが基本
- JOIN: 正規化で分けたテーブルを外部キーで結合して取得する

## 03/07
### Git / GitHub 基礎
- Git: 自分のPCで「変更履歴」を記録するツール（録画機）
- GitHub: Gitの履歴をネット上で共有・管理する場所（YouTube）
- 基本サイクル: add（準備） -> commit（保存） -> push（アップロード）
- ブランチ: 本流（main）を壊さずに作業するための「枝分かれ」
- プルリクエスト (PR): 変更を本流（main）に合流させるための「提案・相談」の場
- マージ: プルリクエストが承認され、変更を本流に反映させること
- pull: GitHub上の最新の状態を自分のPCに「引き寄せる」こと

### GitHub の主要機能
- Issues: バグ報告・機能提案をチケットとして管理する場所。ラベル（bug, enhancementなど）で分類できる
- IssueとPRの連携: PR説明文に `Closes #3` と書くと、マージ時にIssueが自動で閉じる
- Code Review: PRの中でコードの特定の行にコメントをつけて議論する仕組み。Approve / Request Changes / Comment の3判定
- GitHub Actions（CI/CD）: pushやPR作成をトリガーに、テストやデプロイを自動実行する仕組み
- CI = pushのたびに自動テスト → 壊れたらすぐ気づく / CD = テスト通過後に自動デプロイ
- ワークフローは `.github/workflows/` にYAMLファイルを置くだけで動く
- YAMLの構成: on（いつ）→ jobs（何を）→ runs-on（どこで）→ steps（手順）
- `npm ci`: npm installより厳密。lock fileに従って正確に再現するCI向けコマンド
- Projects: Issuesをカンバンボード（Todo / In Progress / Done）で管理し、進捗を可視化する
- Fork: 他人のリポジトリを自分のアカウントにコピーして改造。OSSへの貢献はこの仕組みで行う
- GitHub Pages: リポジトリの内容を無料でWebサイトとして公開できる
- Releases: v1.0のようにバージョンの区切りを公開する仕組み
- 無料枠: パブリックリポジトリは無制限。プライベートは月2,000分まで

## 03/08
### ESLint 警告の修正（StudyTodo実践）
- CIで35件のwarningが検出 → 全件修正して0件に
- warningは「動くけど品質的に問題がある」コード。放置するとバグの温床になる

### ESLint警告の3つのカテゴリ
- **未使用import/変数（no-unused-vars）**: importしたが使っていないモジュールや、宣言したが使っていない変数
  - 例: `import { Shield } from "lucide-react"` → Shieldをどこにも使っていない → import行から削除
  - 例: `const tc = useTranslations("common")` → tcをどこにも使っていない → 行ごと削除
  - **なぜ問題か**: 不要なコードは可読性を下げ、バンドルサイズが無駄に増える
- **不要なeslint-disableディレクティブ**: ESLintのルールを抑制するコメント（`// eslint-disable-next-line`）だが、もう対応するルールの警告が出ていない
  - 例: `// eslint-disable-next-line react-hooks/set-state-in-effect` → この行でルール違反がもう発生していない → コメント削除
  - **なぜ問題か**: 不要な抑制コメントがあると、将来本当に問題が起きたときに見逃す可能性がある
- **exhaustive-deps**: ReactのuseEffect内で使っている変数が依存配列に含まれていない
  - 例: `useEffect(() => { setCurrentMonth(selectedDate); }, [selectedDate])` でcurrentMonthも参照しているが依存配列にない
  - **なぜ問題か**: 依存配列に入っていない変数が変わってもuseEffectが再実行されず、古い値を使ってしまう（ステールクロージャ）

### exhaustive-depsの3つの対処法
1. **依存配列に追加**: 最も正しい対処。変数が変わったらuseEffectも再実行される
2. **eslint-disableで抑制**: 追加するとループが発生する場合に使う。コメントで理由を書くべき
3. **useCallbackでメモ化**: 関数が依存配列に入る場合、useCallbackでラップして参照を安定させる

### `_`プレフィックスパターン
- 将来使う予定だが今は未使用の変数に `_` を付ける慣習
  - 例: `const _handleSocialLogin = async (...) => {...}` → SNSログイン機能は実装済みだがUI側でまだ呼んでいない
- ESLint設定で `varsIgnorePattern: "^_"` を追加すると、`_`始まりの変数は未使用でも警告されなくなる
- `argsIgnorePattern: "^_"` は関数の**引数**に対してのみ有効。**変数宣言**には `varsIgnorePattern` が必要

### ESLint設定ファイル（eslint.config.mjs）のカスタマイズ
- `rules`オブジェクトで各ルールのON/OFF/レベル（error/warn/off）を設定できる
- `"@typescript-eslint/no-unused-vars": ["warn", { argsIgnorePattern: "^_", varsIgnorePattern: "^_" }]`
  - `"warn"` = 警告として報告（ビルドは通る）。`"error"` にするとビルドが止まる
  - `argsIgnorePattern` = 無視する引数の正規表現パターン
  - `varsIgnorePattern` = 無視する変数の正規表現パターン

### 学んだこと
- CIのlint警告は「動くからいい」ではなく、積極的に0件を維持すべき（壊れ窓理論）
- 警告が溜まると本当の問題が埋もれて見えなくなる → 常にクリーンな状態を保つ
- `eslint-disable` は最終手段。使う場合は必ず理由をコメントに書く
- exhaustive-depsは「なぜこの変数を除外するのか」を理解してから抑制する
